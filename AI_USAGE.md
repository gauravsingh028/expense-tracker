# AI Usage Log

## AI Tools Used
- **Antigravity (Google DeepMind)**: Core AI assistant used for generating boilerplate code, React component structures, styling (CSS glassmorphism), and writing the data cleaning logic.

## Key Prompts Used
1. "Build a React-based Splitwise-like Expense Sharing Dashboard that visualizes the spreadsheet data. The app will feature a state-of-the-art, dynamic design (glassmorphism, smooth animations)."
2. "Create an ingestion pipeline to read the CSV file, handle messy date formats like '########' and '15/03/202', normalize text fields like names, and generate an import report."
3. "Write utility functions to calculate debt balances accurately and optimize the debts (e.g. who owes whom)."

## Concrete Cases of AI Errors and Corrections

### Case 1: Extra HTML Tags in JSX (Syntax Error)
- **What happened:** When the AI was asked to update `App.jsx` to inject the `AddExpenseModal`, it performed a multi-replace file content operation that accidentally introduced an extra `</div>` closing tag immediately before the `</main>` tag, breaking the JSX compilation.
- **How it was caught:** The development server/build step failed with a JSX parsing error. I ran `npm run build` to verify the codebase, which would have caught it if it wasn't noticed manually during the diff review.
- **What was changed:** I used the file replacement tool to target the specific lines containing the duplicate `</div>` and removed it, allowing the app to compile properly.

### Case 2: Ignoring Zero and Negative Amounts
- **What happened:** The initial data processing script generated by the AI naively accepted `0` amounts and negative amounts (`-30` for Parasailing refund), allocating negative debt and complicating the total expense calculation.
- **How it was caught:** Looking at the "Total Group Expenses" metric on the dashboard, it didn't align with the expected sum, and people were "owing" negative money.
- **What was changed:** I instructed the AI (and implemented in the data parser) to flag amounts <= 0 as an anomaly and skip processing them as standard expenses, putting them in the import report instead.

### Case 3: Overlooking Settlement Row Math
- **What happened:** The AI generated the `calculateBalances` logic assuming every row was an expense divided among members. It treated "Rohan paid Aisha 5000" (a settlement) as an expense that Rohan paid, giving Rohan an extra 5000 in "Total Spent".
- **How it was caught:** The visual dashboard showed Rohan as the top spender with heavily inflated numbers, and the balance did not cancel out properly between Rohan and Aisha.
- **What was changed:** Updated the logic to check if `split_type` is completely empty or explicitly marked as a settlement in the notes, processing it as a direct reduction in debt rather than a group expense.
